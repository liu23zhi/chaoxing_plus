# chaoxing_plus 重构设计：单平台版 ocsjs 超星实现

日期：2026-05-02

## 1. 背景与目标

当前 `chaoxing_plus` 是一个参照 `ocsjs` 思路开发的超星学习通专用浏览器扩展，采用 Manifest V3 扩展结构，已有视频、学习、答题和面板等能力，但整体范围、调度模型和基础设施仍明显轻于 `ocsjs` 的超星实现。

`ocsjs` 的超星模块已经形成一套完整的单平台自动学习体系，核心集中在 `packages/scripts/src/projects/cx.ts` 及其依赖链上，覆盖：

- 超星页面环境准备与路由匹配
- 课程学习调度
- 视频 / 音频自动学习
- PPT / 书籍 / 长时阅读
- 章节测试
- 作业 / 考试答题
- 旧版 / 新版页面跳转与考试预览兼容
- 倍速限制绕过
- 复制粘贴限制解除
- 人脸识别等待与通知
- 加密字体识别
- 章节与任务点分析工具

本次重构的目标不是在现有扩展架构里继续做轻量增强，而是把 `chaoxing_plus` 重构为一个**内部结构和运行方式尽量贴近 `ocsjs` 超星模块、但只保留超星平台功能**的仓库。

## 2. 约束与明确决策

基于已确认的需求，本次重构遵循以下约束：

1. 以 `ocsjs` 风格为最高优先级，而不是保留现有 `chaoxing_plus` 的扩展式组织方式。
2. 功能范围尽量完整承接 `ocsjs` 的超星部分，而不是只迁移一部分核心能力。
3. 非超星平台代码全部移除，不保留多平台兼容壳层。
4. 不以兼容当前 `chaoxing_plus` 的 UI、配置结构或用户习惯为目标。
5. 允许项目整体改造成更接近 `ocsjs` 的脚本型架构，不必保留 Manifest V3 扩展形态。
6. 答题能力按 `ocsjs` 风格重构，不继续以现有单一外部题库 API 模式作为主设计中心。

## 3. 总体设计结论

最终产物应当是：

> 一个单平台、仅保留超星功能、内部设计尽量等同于 `ocsjs` 超星脚本部分的脚本仓库。

也就是说，仓库名仍可保留为 `chaoxing_plus`，但项目的主心骨将不再是当前的浏览器扩展壳，而是类似 `ocsjs/packages/scripts` 的单项目脚本运行结构。核心业务入口不再围绕 `src/content/index.ts` 的页面识别与多个 manager 分发，而是回到 `CXProject` 驱动模型：由超星项目自身声明域名、页面匹配、脚本注册、调度与业务流程。

## 4. 功能边界

### 4.1 保留的功能

本次重构将尽量保留 `ocsjs` 超星模块中真正属于超星自动学习体系的功能，包括但不限于：

- `CXProject` 及其超星域名匹配逻辑
- 环境准备脚本（真实 top 寻找、iframe 环境修正等）
- 使用提示脚本
- 课程学习脚本
- 课程学习调度器
- 视频 / 音频自动学习
- PPT / 书籍 / 长时阅读任务处理
- 章节测试自动答题
- 作业 / 考试自动答题
- 自动阅读脚本
- 页面自动切换 / 版本切换 / 考试整卷预览相关脚本
- 倍速限制绕过
- 复制粘贴限制解除
- 加密字体 / 繁体字识别
- 人脸识别检测、等待与通知
- 章节分析、任务点搜索、任务完成与跳转逻辑
- 与超星答题、学习流程直接相关的日志、提示、结果呈现与缓存能力

### 4.2 删除的功能

以下内容将从目标架构中移除：

- 所有非超星平台 project
- 多平台入口分发与平台切换逻辑
- 仅为多平台存在的配置项和 UI 文案
- 当前 `chaoxing_plus` 中仅为 MV3 扩展形态存在的 popup / background / 扩展设置页中心结构
- 与最终超星单平台脚本形态无关的旧实现模块

### 4.3 适配而非照搬的部分

虽然目标是尽量贴近 `ocsjs`，但不会机械复制整个仓库。凡是依赖原始多平台基础设施、GM 运行时、跨平台 UI 容器或公共抽象的部分，都需要在 `chaoxing_plus` 中重建**单平台最小等价运行时**，而不是简单复制一个残缺依赖链。

## 5. 目标架构

### 5.1 架构原则

重构后的仓库遵循以下原则：

- **项目驱动，而非页面驱动**：由 `CXProject` 统一组织超星脚本注册与执行。
- **单平台最小运行时**：只保留支撑超星脚本所必需的公共层。
- **功能尽量完整，基础设施尽量收缩**：保留超星业务能力，剔除多平台外壳。
- **运行流贴近 `ocsjs`**：学习调度、任务搜索、答题流程、跳章逻辑和兼容补丁均按 `ocsjs` 主线组织。

### 5.2 建议目录结构

建议将 `chaoxing_plus` 重构为接近如下结构：

```text
chaoxing_plus/
├─ src/
│  ├─ index.ts
│  ├─ projects/
│  │  ├─ cx.ts
│  │  ├─ common.ts
│  │  └─ background.ts
│  ├─ core/
│  │  ├─ answer-wrapper/
│  │  ├─ worker/
│  │  └─ utils/
│  ├─ utils/
│  │  ├─ configs.ts
│  │  ├─ study.ts
│  │  ├─ work.ts
│  │  ├─ render.ts
│  │  └─ index.ts
│  ├─ runtime/
│  │  ├─ project.ts
│  │  ├─ script.ts
│  │  ├─ store.ts
│  │  ├─ message.ts
│  │  ├─ cors.ts
│  │  └─ gm.ts
│  └─ elements/
│     └─ search.infos.ts
├─ package.json
├─ tsconfig.json
└─ README.md
```

该结构不要求完全逐字复制 `ocsjs`，但职责边界应尽量相似：

- `src/index.ts`：脚本总入口，初始化最小运行时并注册超星项目。
- `src/projects/cx.ts`：超星核心项目定义，尽量承接 `CXProject` 风格。
- `src/projects/common.ts`：单平台版公共项目层，只保留超星需要的通用能力。
- `src/projects/background.ts`：仅保留超星所需的调试、日志或后台辅助。
- `src/core/*`：答题器、题目解析、worker、请求与工具层。
- `src/utils/*`：配置、学习辅助、答题辅助、渲染辅助。
- `src/runtime/*`：为了支撑 `cx.ts` 运行而补齐的最小运行时接口层。

### 5.3 计划替换或删除的现有结构

以下现有结构不再作为重构后的核心：

- `src/background/index.ts`
- `src/popup/*`
- `src/content/index.ts`
- `src/content/panel/panel.ts`
- `src/shared/constants.ts` 这一套围绕扩展弹出页的设置组织方式
- 现有按 `video.ts / study.ts / work.ts` 切分、再由内容脚本总入口分发的主流程

这些模块可以在迁移初期作为参考或过渡，但最终不应继续作为主架构存在。

## 6. 运行流程设计

### 6.1 启动流程

新的启动流程如下：

1. 入口初始化单平台脚本运行时。
2. 注册并激活 `CXProject`。
3. `CXProject` 根据当前 URL 和页面状态匹配内部脚本。
4. 各脚本按生命周期执行环境准备、页面提示、学习调度、答题或兼容补丁。

也就是说，真正的调度中心从现有 `src/content/index.ts` 转移到 `src/projects/cx.ts`。

### 6.2 学习流程

课程学习主链路尽量贴近 `ocsjs`：

1. 环境准备脚本先处理可操作的 top 窗口、跨层 iframe、旧版 / 新版页面、必要页面补丁。
2. 学习调度器定位章节列表、未完成任务和当前选中状态。
3. 进入 `study()` 主流程。
4. `study()` 通过 `searchJob()` 在多层 iframe 中搜索任务点。
5. 任务点被识别为视频、章节测试、PPT / 书籍 / 长时阅读、链接任务等类型。
6. 各类型任务交给 `JobRunner` 或章节 / 作业答题器执行。
7. 当前页面任务全部完成后，根据配置决定跳下一小节、下一未完成章节，或等待手动继续。

### 6.3 作业 / 考试流程

作业与考试不再沿用现有轻量 DOM 遍历 + 单一 API 查询模式，而是回到 `ocsjs` 的统一答题器思路：

- 题目解析
- 题型识别
- 搜题与缓存查询
- 答案匹配
- 作答结果汇总
- 保存 / 提交前等待
- 随机兜底与提示
- 结果面板更新

这意味着 `workOrExam()`、章节答题 worker、结果处理和缓存机制都会成为重构的重点。

## 7. 运行时与依赖适配

### 7.1 需要补齐的最小运行时

`cx.ts` 及其依赖链在 `ocsjs` 中依赖大量基础设施，因此本仓库至少需要重建以下能力的单平台版本：

- `Project` / `Script` 风格的脚本注册与生命周期抽象
- 配置读取、持久化与变更监听
- 通知、模态提示、日志输出
- 结果面板与必要渲染容器
- 搜题缓存与作答结果缓存
- 顶层函数桥接 / 跨 iframe 调用
- 类似 GM / unsafeWindow 的能力适配
- 请求工具与跨域请求包装

### 7.2 风险分级

#### 低风险：纯超星业务迁移

- `CXAnalyses`
- `searchJob()`
- `study()` 主流程
- 长时阅读处理
- 章节跳转逻辑
- 旧 / 新页面跳转策略

这些主要依赖 DOM 和超星页面结构，只要运行时接口齐备，迁移风险相对可控。

#### 中风险：运行时耦合能力

- 配置系统
- 通知与弹窗
- 结果面板
- 答题缓存
- 脚本生命周期管理

这些能力会在 `cx.ts` 中高频调用，建议先建立兼容层，再迁业务逻辑。

#### 高风险：环境绑定能力

- `unsafeWindow` / 页面对象访问
- 顶层跨域调用
- 复制粘贴限制解除
- 倍速限制绕过与 `Ext/videojs` override
- 人脸识别等待
- 加密字体识别
- 其他直接依赖页面脚本环境的 hack 能力

这些能力需要依据最终脚本运行形态做适配，是最容易在迁移后出现行为偏差的部分。

## 8. 迁移顺序

本次重构按四个阶段实施：

### 阶段一：重建骨架

- 将现有扩展壳视为待替换对象
- 建立新的脚本型入口
- 搭建单平台公共层和最小运行时接口
- 让 `CXProject` 有可承载的结构位置

### 阶段二：迁移超星主链路

优先迁移：

- `env`
- `guide`
- `study`
- `studyDispatcher`
- `work`
- `autoRead`
- `pageRedirect`
- `versionRedirect`
- `examRedirect`

并打通以下核心方法链：

- `study()`
- `searchJob()`
- `JobRunner`
- `CXAnalyses`
- `workOrExam()`

### 阶段三：补齐高耦合能力

- 配置系统
- 结果面板
- 通知与日志
- 搜题缓存与作答缓存
- 字体识别
- 人脸识别等待
- 倍速限制绕过
- 复制粘贴限制解除

### 阶段四：清理与定型

- 删除旧扩展目录和无效依赖
- 统一命名、构建和配置
- 确认仓库整体形态稳定
- 输出最终 README / 使用说明（如有需要）

## 9. 错误处理策略

错误处理尽量贴近 `ocsjs` 风格，不做“静默失败”的轻量实现。

### 9.1 关键原则

- 所有关键节点都应进入统一日志与提示通道。
- 可降级继续时，优先跳过 / 等待 / 提示手动处理，而不是直接终止全流程。
- 不能安全继续时，应停在当前步骤并明确说明原因。

### 9.2 重点覆盖的错误点

- 页面匹配失败
- 任务点未识别
- 视频元素未找到
- 章节测试未取到题目
- 搜题失败
- 提交前校验失败
- 跨 iframe 访问失败
- 页面重定向未生效

## 10. 验证与验收标准

### 10.1 基础验证

- TypeScript 类型检查通过
- 构建脚本可运行
- 关键目录和模块边界符合目标架构

### 10.2 行为验证主线

至少覆盖以下五条主线：

1. 课程学习调度
2. 视频任务
3. PPT / 阅读 / 长时阅读任务
4. 章节测试
5. 作业 / 考试

### 10.3 高风险兼容点验证

额外单独验证：

- 旧版页面跳转
- 新版考试整卷预览
- 人脸识别等待
- 长时阅读任务
- 加密字体识别
- 倍速限制绕过
- 复制粘贴限制解除

### 10.4 结构验收标准

重构完成时，应满足：

- 仓库不再以 MV3 扩展壳作为主中心架构
- 项目主心骨变为单平台版 `CXProject` + 必要运行时兼容层
- 超星主链路的结构和运行方式尽量贴近 `ocsjs`
- 非超星平台与无关基础设施不再保留在目标架构中

## 11. 非目标

以下内容不属于本次重构目标：

- 保留 `chaoxing_plus` 现有扩展 UI、popup 或设置页习惯
- 兼容旧版 `chaoxing_plus` 配置结构
- 保留多平台抽象壳层，以便未来再扩展其他平台
- 在现有 MV3 代码基础上做最小补丁式增强

## 12. 结论

本次重构的本质不是“增强现有超星扩展”，而是：

**将 `chaoxing_plus` 重构为一个内部结构尽量等同于 `ocsjs` 超星部分的单平台脚本仓库，以 `CXProject` 为中心保留超星全部核心能力，只移除多平台外壳和非超星内容。**

后续实施计划应当围绕这个目标展开，而不是围绕原有扩展结构做局部修补。
